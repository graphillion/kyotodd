# KyotoDD Review (2026-03-21 #13)

`KNOWN_ISSUES.md` と既存の `review*.md` にある論点は除外しています。対象は `BDDV`, `ZBDDV`, `MLZBDDV`, `BtoI`, `CtoI`, `SOP`, `SOPV` の実装・テスト・C++ API ドキュメントです。

[x] **[HIGH] `BDDV(const BDD&, int)` が system variable を拒否せず、`「全要素が同じ BDD」` というコンストラクタ契約を壊します。**  
該当: `include/bddv.h:70-75`, `docs/api/cpp/ops_bddv.rst:91-96`, `src/bddv.cpp:73-92`, `src/bddv.cpp:108-120`, `test/test_bddv.cpp:65-99`。  
長さ 1 のコンストラクタは `src/bddv.cpp:65-70` で system variable を明示的に拒否しているのに、長さ付きコンストラクタは同じ検査をしていません。その結果 `BDDV_Init(); BDDV v(BDDvar(1), 2);` がそのまま通り、`GetBDD(0)` は `BDD(0)`、`GetBDD(1)` は `BDD(1)` になりました。ヘッダ/ドキュメントが約束している「すべての要素が同じ `f`」にはなっておらず、しかも `f` 自体も `must not contain system variables` 契約違反です。  
既存テストは通常の定数・通常変数・長さ境界しか見ておらず、system variable 混入ケースを一切検証していません。

[x] **[HIGH] `CtoI` の item/system 判定が壊れており、正しい item 依存関数を `定数でも boolean でもない何か` と誤判定します。**  
該当: `include/ctoi.h:96-106`, `docs/api/cpp/ops_ctoi.rst:79-93`, `src/ctoi.cpp:111-147`, `src/ctoi.cpp:569-620`, `test/test_ctoi.cpp:724-776`。  
`TopItem()`, `TopDigit()`, `IsBool()` はすべて `BDD_LevOfVar(v)` を `BDD_TopLev() - BDDV_SysVarTop` と比較して item/system を判定していますが、これは「絶対 level」と「ユーザ変数数」を比較しており次元が違います。`BDDV_Init(); bddvar user = BDDV_NewVar(); CtoI c(ZDD::singleton(user));` を試すと、`TopItem()==0`, `IsBool()==0`, `IsConst()==1` になり、純粋な item-only / 0-or-1 の関数を定数扱いしました。`FilterThen/Else` など下流メソッドも同じ誤判定に依存しています。  
しかも `test/test_ctoi.cpp:724-776` の「With item variables」群は `BDD_VarOfLev(1)` / `BDD_VarOfLev(2)` を item variable だと思って使っていますが、そこは system variable です。つまり既存テストは壊れている item-path を実際には一度も通していません。

[x] **[HIGH] `SOP_NewVar` / `SOP_NewVarOfLev` は現在の変数数の parity に依存しており、非 SOP 変数を先に 1 個作るだけで odd VarID を返します。**  
該当: `include/sop.h:243-257`, `src/sop.cpp:35-46`, `test/test_sop.cpp:27-34`。  
実装は「`BDD_NewVar()` を 2 回呼べば後者が even VarID」という前提ですが、変数番号は単純な通番なので、その前に `BDDV_NewVar()` や `BDD_NewVar()` が奇数個入ると parity が反転します。実際に `BDDV_Init(); BDDV_NewVar(); int v = SOP_NewVar();` では `v == 23` になり、直後の `SOP(1).And1(v)` は `SOP::And1: variable must be even` で失敗しました。  
ヘッダは戻り値を無条件に「The even VarID」と約束しており、`SOPV_NewVar()` / `SOPV_NewVarOfLev()` も同じ壊れた API をそのまま公開しています。既存テストは fresh-init 直後の happy path しか見ていません。

[x] **[HIGH] `MLZBDDV` コンストラクタは phase 2 の overflow で例外ではなく `std::exit(1)` を呼び、ライブラリ呼び出し側を巻き込んでプロセスごと終了させます。**  
該当: `src/mlzbddv.cpp:117-149`, `docs/api/cpp/ops_mlzbddv.rst:45-54`, `test/test_mlzbddv.cpp:166-212`。  
同じ phase 2 でも、出力 `i` 自身の置換失敗は `throw std::overflow_error(...)` なのに、他出力 `j` への適用失敗だけ `std::cerr << ...; std::exit(1);` になっています。これは C++ ライブラリとしての失敗モデルが一貫しておらず、呼び出し側が例外で回復する余地を完全に奪います。  
ドキュメントは通常のコンストラクタとしてしか説明しておらず、既存テストも成功ケースしか見ていないため、この fail-stop 経路は CI から漏れています。

[x] **[MEDIUM] `BtoI::operator==` / `operator!=` は数値等価ではなく内部 `BDDV` の構造比較で、公開 API の `BtoI_EQ` と食い違います。**  
該当: `include/btoi.h:58-59`, `include/btoi.h:137-149`, `src/btoi.cpp:55`, `src/btoi.cpp:147-154`, `test/test_btoi.cpp:62-66`, `test/test_btoi.cpp:279-288`。  
`BtoI(const BDDV&)` は `Sup()` を掛けずにそのまま保持する一方、`operator==` は `_bddv == _bddv` の構造比較です。したがって `BtoI a(0); BtoI b(BDDV(BDD(0), 2));` では両方とも整数 0 なのに `a == b` は偽、しかし `BtoI_EQ(a, b)` は 1 になります。  
ヘッダ上はどちらも単に「Equality」としか書かれておらず、既存テストも canonical な表現しか作っていないため、この二重基準が露出していません。

[x] **[MEDIUM] `SOPV::Cube()` / `SOPV::Lit()` は出力ごとに合計せず、一度 `union` してから数えるため、同じ cube が複数出力に現れると過少カウントします。**  
該当: `include/sop.h:353-357`, `docs/api/cpp/ops_sop.rst:322-328`, `src/sop.cpp:775-796`, `src/sop.cpp:919-925`, `test/test_sop.cpp:799-811`。  
現在の実装は各出力の `SOP` を `total += s` で ZBDD の和にまとめ、その後 `total.Cube()` / `total.Lit()` を返しています。これは「各出力の cube/literal 数の総和」ではなく「全出力を union した family の cube/literal 数」です。実際に同じ 1-cube SOP を 2 出力へ入れると `Cube()==1`, `Lit()==1` になり、2 出力ぶんを数えません。  
この誤差は `PrintPla()` の `.p` ヘッダにもそのまま伝播します。既存テストは `xa` と `xb` のように出力ごとに異なる cube しか使っていないので見逃しています。

[x] **[MEDIUM] `ZBDDV::PrintPla()` と `SOPV::PrintPla()` は出力 0 を `~` で書いており、`PLA format` を名乗る API として不正なトークンを出力します。**  
該当: `include/zbddv.h:159-162`, `docs/api/cpp/ops_zbddv.rst:147-149`, `src/zbddv.cpp:341-345`, `src/zbddv.cpp:391-397`, `test/test_zbddv.cpp:392-410`, `include/sop.h:392-396`, `docs/api/cpp/ops_sop.rst:356-358`, `src/sop.cpp:860-868`, `test/test_sop.cpp:694-701`, `test/test_sop.cpp:865-874`。  
`ZBDDV` は定数出力だけ `0/1` を出すのに、非定数ケースでは false 出力を `~` に切り替えています。`SOPV` も同じく false 出力を `~` で書きます。実際に `ZBDDV(ZDD::singleton(v), 0) + ZBDDV(ZDD::singleton(v), 2)` は `1 1~1`、`SOPV` の同等ケースは `1- 1~1` を出力しました。  
ヘッダ/ドキュメントはどちらも単に「PLA format」と説明しており、既存テストは `output.empty()` を見るだけでトークン妥当性を検証していません。

[x] **[LOW] `MLZBDDV` コンストラクタは通常の成功経路でも `_sin` を `stdout` に垂れ流し、API として予想外の副作用があります。**  
該当: `src/mlzbddv.cpp:121`, `docs/api/cpp/ops_mlzbddv.rst:45-54`, `test/test_mlzbddv.cpp:235-249`。  
phase 2 で kernel を 1 個抽出するたびに `std::cout << _sin << std::flush;` を実行しており、構築しただけで標準出力が汚れます。これは `Print()` とは別の hidden side effect で、CLI ツールや機械可読な出力を組み立てる呼び出し側にとってはノイズです。  
ドキュメントは constructor を純粋な構築処理としてしか説明しておらず、既存テストも `Print()` の出力しか見ていないため、この挙動は仕様化も監視もされていません。
