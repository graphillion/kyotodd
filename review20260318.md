## バグリスク / 正当性

- [x] **`bddfinal()` が `bdd_node_count` と `bdd_node_max` をリセットしない** (`src/bdd_base.cpp:178-203`)
  `bddfinal()` は `bdd_node_used` をリセットするが、`bdd_node_count` と `bdd_node_max` は古い値のまま残る。`bdd_nodes` が `nullptr` に設定された後も、`bdd_node_count` が非ゼロのままであるため、`bddfinal()` 後に `bddinit()` を呼ばずライブラリ状態を参照するコードがあれば矛盾が生じうる。

- [x] **`bddfinal()` が `bdd_gc_threshold` をリセットしない** (`src/bdd_base.cpp:178-203`)
  `bddgc_setthreshold()` で変更した閾値が `bddfinal()` → `bddinit()` を経てもデフォルト値 (0.9) に戻らない。`bddinit()` も `bdd_gc_threshold` を再設定しない。ユーザーが再初期化後にデフォルト状態を期待すると予期せぬ動作になる。

- [x] **`ZDD::ImplySet`, `ZDD::SymSet`, `ZDD::CoImplySet` が `int` を引数に取る** (`include/bdd_types.h:604,610,612`)
  対応する C 関数 (`bddimplyset`, `bddsymset`, `bddcoimplyset`) は `bddvar` (uint32_t) を受け取るが、ZDD クラスのメンバ関数は `int` を受け取る。負の値が渡されると `int` → `uint32_t` の暗黙変換により非常に大きな変数番号に変化し、範囲チェックで例外が投げられる（未定義動作ではない）が、API 一貫性として `bddvar` に統一すべき。

- [x] **`node_index` 等のインライン関数にリリースビルドでの実行時境界チェックがない** (`include/bdd_internal.h:58-64`)
  `BDD_DEBUG_ASSERT` はリリースビルドで無効化される。不正なノード ID が渡された場合、`bdd_nodes` の範囲外アクセスが発生し未定義動作になる。内部関数であり呼び出し元が正当性を保証する設計だが、防御的プログラミングの観点から実行時チェックの追加を検討すべき。
  → **対応不要**: ホットパスの内部関数であり、リリースビルドでの実行時チェックはパフォーマンスに大きな影響がある。`BDD_DEBUG_ASSERT` による開発時チェックで十分。

- [x] **`export_collect` が再帰を使用しスタックオーバーフローのリスクがある** (`src/bdd_io.cpp:19-28`)
  DAG の深さが大きい場合、再帰呼び出しでスタックを使い切る可能性がある。`bddgc_mark` (bdd_base.cpp:407) は明示的なスタック（`std::vector`）を使用しているのと対照的。

- [ ] **`bdd_check_reduced_rec` が再帰を使用しスタックオーバーフローのリスクがある** (`src/bdd_base.cpp:739-746`)
  `export_collect` と同様、深い DAG でスタックオーバーフローの可能性がある。`BDD_RecurGuard` による深さ制限もない。

## Python バインディング

- [ ] **例外翻訳にフォールバックがない** (`python/src/kyotodd/_binding.cpp:22-31`)
  `overflow_error`, `invalid_argument`, `out_of_range` のみ捕捉している。以下が欠落:
  - `std::logic_error` → 未実装関数 (`PrintPla`, `ZLev`, `SetZSkip`) や廃止関数 (`bddisbdd` 等) が投げる
  - `std::bad_alloc` → メモリ不足時
  - `std::exception` → 汎用フォールバック
  捕捉されない C++ 例外は Python 側でクラッシュ (`terminate`) を引き起こす可能性がある。

- [ ] **`g_initialized` フラグがスレッドセーフでない** (`python/src/kyotodd/_binding.cpp:9`)
  `static bool g_initialized` にはミューテックスやアトミック保護がない。Python の GIL が通常は保護するが、`nogil` 拡張や複数インタープリタの場合に競合が発生する。

- [ ] **`ensure_init()` が暗黙的に初期化を行う** (`python/src/kyotodd/_binding.cpp:11-16`)
  ユーザーが `init()` を呼ばなくてもデフォルト引数で `bddinit(256, UINT64_MAX)` が呼ばれる。ユーザーが `node_max` を制限したいケースで、意図せずデフォルト初期化されてしまう可能性がある。少なくとも初期化済みかどうかの確認ログまたは警告が望ましい。

- [ ] **`finalize()` 後の Python 側 BDD/ZDD オブジェクトのダングリング** (`python/src/kyotodd/_binding.cpp:50-55`)
  `finalize()` は `bddfinal()` を呼び、全ノードを解放する。しかし Python 側に BDD/ZDD オブジェクトが残っている場合、それらの `root` はダングリングポインタ同然になる。デストラクタ (`bddgc_unprotect`) が呼ばれると無効なポインタを `gc_roots` から除去しようとするが、`gc_roots` は `bddfinal()` で `clear()` 済みなので `erase` は空振りになる。直接的なクラッシュにはならないが、`finalize()` 後に既存オブジェクトを操作するとノードアクセスで未定義動作になる。

## ビルド設定

- [ ] **コンパイラ警告フラグが設定されていない** (`CMakeLists.txt`)
  `-Wall -Wextra` が設定されていない。暗黙の型変換、未使用変数、符号変換などの警告が検出されない。`target_compile_options` でライブラリとテストの両方に設定すべき。

- [ ] **Debug/Release ビルド構成の区別がない** (`CMakeLists.txt`)
  `CMAKE_BUILD_TYPE` による最適化レベルの切り替えがない。BDD/ZDD ライブラリは計算集約型であり、`-O2`/`-O3` の有無でパフォーマンスに大きな差がある。

- [ ] **Python バインディングに `-fvisibility=hidden` が設定されていない** (`python/CMakeLists.txt`)
  pybind11 モジュールではシンボル可視性の制限が推奨される。設定がないと、バイナリサイズの増大やシンボル衝突のリスクがある。

- [ ] **`bddvdump` の引数が `const bddp*` でない** (`include/bdd_io.h:160`)
  `void bddvdump(bddp *p, int n)` は配列を読むだけなので `const bddp *p` であるべき。

- [ ] **I/O 関数の配列長パラメータが `int`** (`include/bdd_io.h:19,34,54,71,92,109`)
  `bddexport`, `bddimport`, `bddimportz` の `lim` パラメータが `int`。大きな配列や `size_t` ベースの API と組み合わせる場合に型不一致が生じる。ただし、実際の使用では `INT_MAX` を超える配列は稀。

- [ ] **キャッシュミス時の戻り値がヘッダで文書化されていない** (`include/bdd_base.h:234-237`)
  `bddrcache` / `bddrcache3` はキャッシュミス時に `bddnull` を返すが、ヘッダのコメントにはその記述がない。実装を読まないと利用方法がわからない。

## コード品質

- [ ] **`bddimply` が `bdd_gc_guard` なしで再帰的に呼ばれる** (`src/bdd_ops.cpp:397-456`)
  `bddimply` は公開ラッパーだが、自身を再帰的に呼び出す（line 447, 452）。ノード生成は行わないため GC 保護は不要だが、`BDD_RecurGuard` のみでパターンが他の公開関数（wrapper + `_rec`）と異なる。一貫性の観点で注意。

- [ ] **`bddispoly` がキャッシュを使用しない** (`src/zdd_adv2.cpp:9-22`)
  `bddispoly` はループで ZDD を走査するが、結果をキャッシュしない。複数回呼び出される場合（例: `bdddivisor` 内で繰り返し呼ばれる）にパフォーマンスが低下する。

- [ ] **`bddsymgrp` / `bddsymgrpnaive` が O(n²) の対称性チェックを行う** (`src/zdd_adv2.cpp:273-350`)
  変数ペアごとに `bddsymchk` を呼ぶため、変数数が多いと計算量が大きくなる。ただし、BDD ベースの対称性検出アルゴリズムの一般的な実装パターン。

- [ ] **静的 `const BDD`/`ZDD` オブジェクトが `bddfinal()` 後にダングリング状態になる** (`src/bdd_class.cpp:6-12`)
  `BDD::False`, `BDD::True`, `ZDD::Empty` 等は静的初期化時に `bddgc_protect` で GC ルートに登録される。`bddfinal()` は `gc_roots().clear()` でこれらの登録を消す。再度 `bddinit()` しても、静的オブジェクトは再登録されない。これらの `root` 値は全て定数ターミナル (bddfalse/bddtrue/bddnull) であるため実害はないが、設計上の注意点。

## ドキュメント

- [ ] **`BDD_RecurLimit = 8192` の根拠が文書化されていない** (`include/bdd_types.h:44`)
  マジックナンバーの選定理由がコメントにない。デフォルトスタックサイズとの関係、または経験的な値かが不明。

- [ ] **`BDDCARD_MAX`, `BDDLIT_MAX`, `BDDLEN_MAX` のサチュレーション動作が文書化されていない** (`src/zdd_adv.cpp:610,670,712`)
  `bddcard`, `bddlit`, `bddlen` は結果が `(1<<39)-1` に達するとサチュレートする。ヘッダのドキュメントにはこの制限が記載されていない。ユーザーが正確な値を期待して `bddcard` を呼び、サチュレートした値を正確な値と誤解する可能性がある。`bddexactcount` の存在もヘッダコメントで案内すべき。
