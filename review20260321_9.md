# KyotoDD BDD Review (2026-03-21 #9)

`KNOWN_ISSUES.md` と既存の `review*.md` にある論点は除外しています。

[x] **[HIGH] `bddprime` 系 API が 65536 変数を超えた時点で使えなくなり、公開仕様の 31-bit variable model と矛盾しています。**
該当: `src/bdd_base.cpp:882-894`, `src/bdd_class.cpp:12-43`, `src/bdd_ops.cpp:605-607`, `src/bdd_ops.cpp:681-682`, `src/bdd_ops.cpp:821-847`, `include/bdd_types.h:20-35`, `docs/concepts.rst:58-69`。
`bddprime()` は `v > 65536` を無条件で拒否しますが、型定義と概念ドキュメントは variable number を 31-bit 値として説明しています。さらに `BDD::prime` / `prime_not` / `cube` / `clause`、単変数版の `Exist` / `Univ`、`bddswap` の内部再構成はすべて `bddprime()` に依存しているため、高番地変数を作成できても標準 API 群のかなりの部分が突然使えなくなります。手元では `bddnewvar()` を 65537 回呼んだ後、`bddprime(65537)` は `"bddprime: auto-expansion beyond 2^16 variables is not supported"` で失敗する一方、`BDD::getnode(65537, bddfalse, bddtrue)` は成功しました。基盤データ構造は受け入れられるのに、literal 構築 API だけが別の上限で塞がれており、既存テストにもこの境界値はありません。

[x] **[MEDIUM] `bddexist(..., std::vector<bddvar>)` / `bdduniv(..., std::vector<bddvar>)` が重複変数を正規化せず、意味的に冗長な入力で余計なノードを作ります。**
該当: `src/bdd_ops.cpp:579-594`, `src/bdd_ops.cpp:597-602`, `src/bdd_ops.cpp:673-678`, `src/bdd_base.cpp:736-746`, `include/bdd_types.h:289-311`, `test/test_bdd.cpp:1462-1548`。
`vars_to_cube()` は入力を sort するだけで unique 化せず、`BDD::getnode_raw()` を直接積んで cube を作ります。そのため `{v1, v1}` のような入力では、checked path なら禁止される「親子が同レベル」の一時 cube ノードを作ってしまいます。手元では `f = x1 & x2` に対して `f.Exist({v1})` と `f.Exist({v1, v1})` は同じ結果を返すのに、`bddused()` は前者の後で 3 のまま、後者の後で 4 に増えました。公開コメントは単に「list of variables」としか言っておらず、重複を reject も dedup もしない現在の実装は、等価な問い合わせで node table を無駄に消費します。既存テストもすべて重複なし vector しか見ていません。
→ 対応: `vars_to_cube()` でソート後に `std::unique` で重複除去を追加。テスト追加済み。

[x] **[MEDIUM] C++ API ドキュメントの Node Creation 節が、存在しない `getnode` / `getznode` 自由関数を参照しています。**
該当: `docs/api/cpp/base.rst:32-40`, `include/bdd_types.h:538-550`, `include/bdd_types.h:1054-1066`, `src/bdd_base.cpp:736-808`。
ドキュメントは `.. doxygenfunction:: getnode` と `.. doxygenfunction:: getznode` を掲げていますが、リポジトリ内の実体は `BDD::getnode(...)` と `ZDD::getnode(...)` のクラスメソッドだけです。`include/` / `src/` を検索しても対応するグローバル関数は見当たりません。結果として、Node Creation セクションは docs site 上で空振りまたは誤誘導になり、利用者がたどるべき checked constructor を隠してしまっています。
→ 対応: `base.rst` から存在しない `getnode` / `getznode` ディレクティブを削除。クラスメンバは `types.rst` の `doxygenclass` で自動出力される。

[x] **[LOW] Python の `BDD.var(v)` ドキュメントと型スタブが、実装よりも厳しい前提を案内しています。**
該当: `docs/api/python/bdd.rst:16-21`, `python/src/kyotodd/_core.pyi:25-33`, `python/src/kyotodd/_binding.cpp:210-214`, `src/bdd_base.cpp:882-894`。
API ドキュメントと `.pyi` は「`v` は事前に作成済みの変数でなければならない」と書いていますが、実装は `BDD.var(v)` から `BDDvar(v)` を経由して `bddprime(v)` を呼び、`v` に達するまで `bddnewvar()` で自動拡張します。しかも `.pyi` では作成関数名が実在しない `newvar()` になっています。実装の本当の契約は「未作成でも自動拡張される。ただし 65536 を超えると別の理由で失敗する」であり、現行ドキュメントではその挙動が正しく伝わりません。
→ 対応: `bdd.rst` と `_core.pyi` の `BDD.var(v)` 説明を「未作成なら自動作成される」に修正。

[x] **[LOW] `support_vec()` のチュートリアル出力例が、実装と既存テストの返却順序と食い違っています。**
該当: `docs/tutorials/bdd_basics.rst:88-90`, `src/bdd_ops.cpp:499-512`, `test/test_bdd.cpp:1301-1306`。
チュートリアルは `f = a & b` に対して `support_vec()` の例を `[1, 2]` と書いていますが、実装は level 降順で sort して返しており、既存テストも default ordering で `[v3, v2, v1]` を正解として固定しています。利用者がチュートリアルの順序をそのまま期待すると、サンプルコードと実挙動が一致しません。
→ 対応: チュートリアルの出力例を `[2, 1] (descending level order)` に修正。
