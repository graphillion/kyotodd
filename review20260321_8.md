# KyotoDD BDD Review (2026-03-21 #8)

`KNOWN_ISSUES.md` と既存の `review*.md` にある論点は除外しています。

[x] **[HIGH] `BDD::uniform_sample()` が `n > bddvarused()` を安全に扱えず、未作成の domain 変数でクラッシュまたは分布破壊を起こします。**
該当: `src/bdd_class.cpp:128-190`, `include/bdd_types.h:449-462`, `test/test_bdd.cpp:7194-7349`。`src/bdd_class.cpp:155-157` は `var2level[1..n]` を無条件に参照しており、変数を 1 個も作らずに `BDD::True.uniform_sample(rng, 3, memo)` を呼ぶと、手元では ASan で `src/bdd_class.cpp:156` の null dereference になりました。さらに変数を 1 個だけ作った状態で `BDD::True.uniform_sample(rng, 5, memo)` を 200 回回しても、2..5 は 1 度も出現せず、未作成の don't-care 変数を完全に無視しました。`count(n)` / `exact_count(n)` は「{0,1}^n 上の代入数」を数える契約なのに、`uniform_sample(n)` だけが同じ空間をサンプルしていません。既存テストは常に `n <= bddvarused()` のケースしか通しておらず、この境界条件が CI から抜けています。
→ 対応: `n > bddvarused()` 時に `invalid_argument` を投げるバリデーションを追加。テスト追加済み。

[x] **[HIGH] `BddCountMemo` が variable-level mapping の変更を検知できず、`bddnewvaroflev()` 後に `exact_count(..., memo)` と `uniform_sample(..., memo)` が誤答を返します。**
該当: `include/bdd_types.h:180-203`, `include/bdd.h:226-240`, `src/bdd_base.cpp:345-371`, `src/bdd_class.cpp:144-147,189-190`, `test/test_bdd.cpp:7051-7078`, `test/test_bdd.cpp:10835-10892`。`BddCountMemo` が保持している識別子は root と `n` だけで、level mapping の世代情報を持っていません。そのため `bddnewvaroflev()` で既存変数の level がずれても、`include/bdd.h:226-240` は memo をそのまま再利用します。手元では `v1` の BDD に対して `exact_count(1, memo)` を一度計算し、その後 `bddnewvaroflev(1)` を挟んで同じ memo を再利用すると、期待値 1 ではなく 0 が返りました。`bddnewvaroflev()` がクリアしているのは内部 op cache だけで、外部 memo は無効化されません。既存テストは「BDD が違う」「`n` が違う」ケースしか見ておらず、公開仕様で認めている variable reordering との組み合わせを全く検証していません。
→ 対応: `BddCountMemo` の Doxygen に `@warning` を追加し、`bddnewvaroflev()` 後は memo を破棄して再作成すべき旨を明記。

[x] **[MEDIUM] `BDD::cube()` / `BDD::clause()` が `INT_MIN` literal で signed overflow の未定義動作に入ります。**
該当: `src/bdd_class.cpp:20-27`, `src/bdd_class.cpp:33-40`, `test/test_bdd.cpp:176-225`, `python/tests/test_bdd_new.py:107-138`。両関数とも負リテラルを `lit < 0 ? -lit : lit` で正規化していますが、`lit == INT_MIN` だけは `-lit` 自体が C++ の未定義動作です。手元で UBSan 付きで `BDD::cube({std::numeric_limits<int>::min()})` を実行すると、`src/bdd_class.cpp:26` で signed overflow が報告されました。その後に `bddprime` が例外を投げるとしても、例外到達前に UB を踏んでいる点が問題です。既存テストは 0 と通常範囲の literal しか見ていません。
→ 対応: `lit == INT_MIN` を UB 発生前に `invalid_argument` で弾くガードを追加。テスト追加済み。

[x] **[MEDIUM] `BDD::XPrint0()` / `BDD::XPrint()` は公開コメント上は「グラフを表示する API」ですが、実際には必ず `logic_error` を投げます。**
該当: `include/bdd_types.h:396-399`, `include/bdd.h:346-351`, `include/bdd_io.h:408-411`, `src/bdd_io.cpp:1913-1921`, `test/test_bdd.cpp:5747-5754`。BDD クラスのコメントは単に `bddgraph0` / `bddgraph` の wrapper と説明していますが、低レベル側は `include/bdd_io.h` で「Always throws」と明記され、実装も無条件で `logic_error` を投げています。さらにテストは「投げること」を正解として固定しており、この API が実用不能であることを仕様として焼き付けています。利用者から見ると、クラス API と低レベル API の説明が食い違っており、Doxygen だけ見ると使える機能に見えてしまいます。
→ 対応: BDD/ZDD の `XPrint0()` / `XPrint()` の Doxygen コメントを `@deprecated Always throws std::logic_error. Use save_graphviz() instead.` に修正。

[x] **[MEDIUM] Python 向け BDD チュートリアルが存在しない `f.size` プロパティを案内しており、サンプルコードをそのまま実行できません。**
該当: `docs/tutorials/getting_started.rst:69-86`, `docs/tutorials/bdd_basics.rst:146-152`, `python/src/kyotodd/_binding.cpp:373-381`, `python/tests/test_bdd.py:109-124`。チュートリアルは `f.size` を Python 側の node count と説明していますが、binding が公開しているのは `raw_size` と `plain_size` だけで、`size` プロパティは定義されていません。したがってドキュメントどおりに `print(f"f has {f.size} nodes")` を実行すると `AttributeError` になります。既存の Python テストも `raw_size` しか参照していないため、チュートリアルの破綻を検出できていません。
→ 対応: `getting_started.rst` と `bdd_basics.rst` の `f.size` を `f.raw_size` に修正。差分テーブルも更新済み。

[x] **[LOW] `BDD::Print()` は公開 sentinel である `BDD::Null` を安全に表示できず、デバッグ用 API のはずが例外で落ちます。**
該当: `include/bdd_types.h:394-395`, `include/bdd.h:336-343`, `src/bdd_base.cpp:400-403`, `test/test_bdd.cpp:5504-5536`。`Print()` は先頭で `bddtop(root)` を呼ぶため、`BDD::Null.Print()` は `bddtop: bddnull` の `invalid_argument` で終了します。`BDD::Null` はクラス定数として公開されており、他の問い合わせ API では `raw_size()==0` や `save_graphviz()` no-op のような fail-safe な経路もあるのに、ログ出力だけが fail-stop です。既存テストは通常ノードと `BDD::False` しか見ておらず、`Null` の表示経路が未カバーです。
→ 対応: 仕様として妥当と判断。`Print()` の Doxygen に `@throws std::invalid_argument if this is BDD::Null` を明記。
