# KyotoDD Review (2026-03-23 #2)

`KNOWN_ISSUES.md` と既存の `review*.md` にある論点は除外しています。対象は `BDDV`, `ZBDDV`, `MLZBDDV`, `BtoI`, `CtoI`, `SOP`, `SOPV` の実装・テスト・C++ API ドキュメントです。

[x] **[HIGH] `BtoI::UpperBound(const BDD&)` / `LowerBound(const BDD&)` は制約 `f` を正しく反映しておらず、到達不能な値を返します。**  
該当: `include/btoi.h:100-103`, `docs/api/cpp/ops_btoi.rst:112-119`, `src/btoi.cpp:485-541`, `test/test_btoi.cpp:415-424`。  
実装は `signUniv = d.Univ(f)` と各ビットの `c0 = (cond & d).Exist(f)` を組み合わせていますが、`f` が指定している「許される代入の集合」をそのまま絞り込みに使えていません。`BDDV_Init(); int v = BDDV_NewVar(); BDD x = BDD::prime(v); BtoI a = BtoI_ITE(x, BtoI(10), BtoI(-3));` を手元で確認すると、無制約では `UpperBound()==10`, `LowerBound()==-3` なのに、`a.UpperBound(x)` / `a.UpperBound(~x)` はどちらも `15`、`a.LowerBound(x)` / `a.LowerBound(~x)` はどちらも `-7` になりました。`x` の下で到達できる値は `10` だけ、`~x` の下では `-3` だけなので、現在の戻り値は API 契約と一致していません。  
既存テストは無制約版しか見ておらず、制約付きオーバーロードを一度も検証していません。

[x] **[HIGH] `BDDV_ImportPla(..., sopf = 1)` は実質的に壊れており、妥当な SOP 互換 PLA を `BDDV(BDD): BDD contains system variables` で拒否します。**  
該当: `include/bddv.h:270-273`, `docs/api/cpp/ops_bddv.rst:239-243`, `src/bddv.cpp:673-716`, `src/bddv.cpp:776-779`, `test/test_bddv.cpp:720-742`。  
`sopf` 指定時は `src/bddv.cpp:675-677` / `693-695` で入力変数を偶数 level に寄せていますが、実際の積項パーサはその後も `src/bddv.cpp:715-716` で `BDDV_SysVarTop + 1 + i` をそのまま入力 level とみなしています。そのため `sopf=1` 経路では入力文字列から組み立てた cube に system variable が混入し、最後の `BDDV(final_onset[j])` で例外になります。  
実際に `.i 1\n.o 1\n1 1\n.e\n` を `BDDV_ImportPla(f, 1)` に渡すと、正常な 1 入力 1 出力 PLA なのに `std::invalid_argument("BDDV(BDD): BDD contains system variables")` が発生しました。既存テストはデフォルトの `sopf=0` しか通していません。

[x] **[MEDIUM] `BtoI::GetInt()`, `StrNum10()`, `StrNum16()` は非定数関数を拒否せず、`変数をすべて 0 に固定した枝` を黙って返します。**  
該当: `include/btoi.h:119-121`, `docs/api/cpp/ops_btoi.rst:165-175`, `src/btoi.cpp:592-617`, `src/btoi.cpp:624-815`, `test/test_btoi.cpp:433-478`, `test/test_btoi.cpp:485-514`。  
`GetInt()` は `Top() > 0` のとき無条件で `At0(Top()).GetInt()` へ降り、`StrNum10()` / `StrNum16()` も `while (f.Top() > 0) f = f.At0(f.Top());` で全変数を 0 側へ潰してから数値化しています。したがって `BtoI bi(BDD::prime(v));` のような非定数値でも、`bi.GetInt() == 0`, `bi.StrNum10(buf) == "0"`, `bi.StrNum16(buf) == "0"` になります。  
この `bi` は `x=0` なら 0、`x=1` なら 1 を表すので、現状の変換 API は「関数の整数値」ではなく「ゼロ割当て時の一枝」を silently に返しており危険です。既存テストは定数ケースしか文字列化しておらず、変数依存ケースでは `At0/At1` による枝の確認しかしていません。

[x] **[MEDIUM] `BtoI_atoi` と `CtoI_atoi` は不正な文字列をエラーにせず、途中までを黙って別の値として受理します。**  
該当: `include/btoi.h:167`, `docs/api/cpp/ops_btoi.rst:222`, `src/btoi.cpp:955-974`, `test/test_btoi.cpp:466-478`, `include/ctoi.h:282`, `docs/api/cpp/ops_ctoi.rst:309`, `src/ctoi.cpp:1028-1079`, `test/test_ctoi.cpp:601-623`。  
`BtoI_atoi` の 2 進経路は各文字を厳密に検証せず、`'1'` 以外を事実上 0 として扱います。10/16 進経路と `CtoI_atoi` 側も `strtol(..., NULL, base)` に依存しており、末尾まで正しく読めたかを確認していません。  
手元では `BtoI_atoi("0b102").GetInt() == 4`, `BtoI_atoi("12x34").GetInt() == 12`, `BtoI_atoi("0x1g").GetInt() == 1`, `CtoI_atoi("0b102").GetInt() == 2`, `CtoI_atoi("12x34").GetInt() == 12` を確認しました。入力検証付きの変換 API だと期待すると危険です。

[x] **[MEDIUM] `BDDV_ImportPla` は不正な積項行を検出しても失敗扱いにせず、正常値に見える all-zero ベクタを返します。**  
該当: `include/bddv.h:270-273`, `docs/api/cpp/ops_bddv.rst:239-243`, `src/bddv.cpp:706-747`, `test/test_bddv.cpp:720-742`。  
`parse_product_term()` は入力パターンや出力パターンが壊れていると `false` を返しますが、その戻り値は `src/bddv.cpp:745-747` で完全に無視されています。結果として、文法エラーのある PLA でも `onset` / `dcset` を一切更新しないまま処理が継続し、最後に「すべて 0 の BDDV」が正規結果のように返ります。  
たとえば `.i 1\n.o 1\n2 1\n.e\n` のように入力側へ不正文字 `2` を含む PLA を渡すと、例外や null ではなく、長さ 2 の `(onset || dcset)` ベクタが両方 `BDD(0)` のまま返りました。これは「構文エラー」と「正しいゼロ関数」が区別できません。

[x] **[MEDIUM] `CtoI::MaxVal()` / `MinVal()` は `CtoI_Null()` を保持できず、エラー sentinel を 0 に潰します。**  
該当: `include/ctoi.h:184-187`, `docs/api/cpp/ops_ctoi.rst:177-183`, `src/ctoi.cpp:644-670`, `test/test_ctoi.cpp:420-432`。  
`MaxVal()` は先頭で `if (_zbdd == ZDD(-1)) return CtoI(0);` としており、null/error 値を通常の数値 0 に変換してしまいます。`MinVal()` も `-((-*this).MaxVal())` に委譲しているため同じ問題を引き継ぎます。  
`CtoI n = CtoI_Null();` を使って確認すると、`n.MaxVal().GetZBDD().GetID()` も `n.MinVal().GetZBDD().GetID()` も `bddnull` ではなく `bddempty` になり、`CtoI(0)` と区別できなくなりました。ほかの集約系 API は null をそのまま返すものが多く、ここだけ失敗状態が消えるのは不整合です。

[x] **[LOW] `MLZBDDV::N_sin()` と `GetZBDDV()` の公開契約が一致しておらず、`N_sin()` は実在しない phase-1 スロットまで数えます。**  
該当: `include/mlzbddv.h:63-66`, `docs/api/cpp/ops_mlzbddv.rst:72-80`, `src/mlzbddv.cpp:43-45`, `src/mlzbddv.cpp:104-108`, `src/mlzbddv.cpp:168-169`, `test/test_mlzbddv.cpp:56-60`, `test/test_mlzbddv.cpp:250`。  
ドキュメントは `N_sin()` を「抽出された sub-expression inputs の数」、`GetZBDDV()` を「`N_out()` 以上の index に sub-expression definition が入る」と説明しています。しかし実装は phase 1 の各出力に対して無条件に `_sin++` する一方、その定義自体は保持していません。実際に phase 2 抽出が 1 回も起きない単純な 1 出力ケースでも `N_out()==1`, `N_sin()==1` なのに、`GetZBDDV().Last()==0` で index 1 の定義は存在しません。  
現在の API だと `N_sin()` を見て `GetZBDDV()` 側の定義数を見積もる呼び出し側が誤ります。少なくとも docs の説明とは一致していません。

[x] **[LOW] `ZBDDV::Mask()` と `SOPV::Mask()` は `start` 以降を 0-based に詰め直さず元の index を保持するため、ドキュメントの「sub-array / sub-range」表現と食い違います。**  
該当: `include/zbddv.h:134`, `docs/api/cpp/ops_zbddv.rst:129-131`, `src/zbddv.cpp:243-257`, `test/test_zbddv.cpp:200-218`, `include/sop.h:376`, `docs/api/cpp/ops_sop.rst:342-344`, `src/sop.cpp:815`, `test/test_sop.cpp:875`。  
実装は `Mask(start, length)` で抜き出した要素を `tmp += ZBDDV(elem, i);` と元の index `i` のまま再配置しています。したがって `ZBDDV zv; zv += ZBDDV(sa, 1); auto mz = zv.Mask(1);` の結果は、`mz.GetZBDD(0)` が空で `mz.GetZBDD(1)` に要素が残る「疎な slice」です。`SOPV::Mask()` も `ZBDDV::Mask()` をそのまま包んでいるので同じ挙動です。  
もし意図が「絶対 index を保ったマスク」なら docs をそう書くべきですし、通常の sub-array 抽出を期待する API なら現実装がずれています。既存テストも `start=0` か、この絶対 index 動作を前提にした確認しかしていません。
